## Problem Statement

When the user has an AI agent write or plan code on their behalf, the agent forgets the design rules, maintainability patterns, and consistency idioms the user cares about — and the user can't catch every miss on review. The result: code that compiles and passes tests but drifts from the user's standards. Over time, the codebase loses coherence not because the user wrote bad code, but because the agent who wrote 80% of it didn't know the rules.

The user's framing in this session:
- *"Set up the agent harness so my code quality is improved."*
- *"More like RAG — a set of code consistency and design rules the agent pulls from as it writes code and plans."*
- *"Cover areas the user forgets such as when making a system maintainable."*
- *"It has to be scalable so it will act like a distributed system of corpuses attacking different rules."*
- *"Let's not add dependencies. Let's think about the design of the problem from scratch."*
- *"Needs a docker container so it's easy to install and run. The container spins up a local container on the user's machine so it can access the data."*

This design starts fresh — no Theia, no Aether dependency, no techdebt dependency, no Aide dependency. The only deliverable is a system that makes any AI agent on the user's behalf produce code that respects the user's standards, shipped as a Docker container so install is a one-liner.

## What Makes This Cool

- **The corpus IS the moat.** Not the IDE, not the agent, not the framework. A growing, curated set of rules that ride along with every agent invocation. The product is what's in the corpus.
- **One Docker command to install.** `docker run -v ~/.betterai:/data -p 7777:7777 betterai/server` and you're live. No npm, no bun, no Node version dance. Pulls the seed corpus on first run. Same install on macOS / Linux / Windows-WSL.
- **Agent-agnostic via MCP.** Built around MCP. Claude Code uses it via its built-in MCP client. claude.ai managed agents use it via remote MCP. Cursor, Aether, Codex, Gemini CLI — any agent that speaks MCP gets the corpus for free.
- **Hybrid rule shape — prose with optional bite.** Most rules are markdown the agent reads. Rules that *can* be checked programmatically *should* be checked programmatically. Verification is opt-in per rule, so the corpus grows organically.
- **Distributed corpus, single API.** Internally the corpus has N sub-corpuses (security, perf, naming, idempotency, observability, …) each with its own retrieval discipline. From any agent's perspective, it's one `retrieve_rules(context)` call.
- **VSCode is the human surface.** Not a custom IDE shell. The user's daily editor gets a sidebar to browse rules, a status bar dot for corpus state, a webview for editing rules, and an audit log of what was retrieved per agent session.

## Constraints

- **Docker container is the install vector.** No npm publish for end users. `docker run` (or `docker compose up`) is the install story. The container mounts the user's corpus dir + reads from the host filesystem for project context.
- **No new code dependencies on prior projects.** BetterAI is its own repo. It does NOT depend on Aether, Aide, GBrain, or techdebt. Each is a separate world; cross-pollination happens via MCP and shared rule files, not code coupling.
- **TypeScript inside the container** (the MCP server). Rule executable checks can be TS, regex, ast-grep, or shell.
- **Local-first.** Corpus on host disk (mounted into the container). Optional embedding sidecar (sqlite-vec or chromadb) inside the container. No SaaS.
- **MCP-native from day one.** The retrieval API is an MCP server. No custom protocol.
- **VSCode is the only human surface in v1.** No web UI. CLI utilities exist (`betterai-cli`) but are minimal.
- **Personal toolkit framing.** Single user, evals serve robustness, no market polish requirement.

### Why Docker is load-bearing

- **One install command.** `docker run` (or `docker compose up`) on macOS, Linux, Windows-WSL. No "what version of Node do I have." The container ships its own runtime.
- **Reproducible.** Same image hash → same MCP server behavior. The embedding model, ast-grep binary, sqlite-vec extension, and all transitive deps are baked into the image.
- **Sandboxed file access.** The container only sees what you mount. Corpus dir is read-write; project code dirs are read-only. Reduces blast radius for `check.kind=shell` rules.
- **Easy reset / wipe / rollback.** `docker rm betterai && docker pull betterai/server:vX && docker run ...`.
- **Same image for personal + (eventual) shared use.** v2 multi-user deploy is `docker compose up` with a tweaked compose file; v1 is the same container in single-user mode.

## The Rule Shape (Hybrid — locked in session)

A rule is a markdown file. YAML frontmatter is optional but recommended.

```markdown
---
id: webhook-handlers-must-dedupe
domain: idempotency
title: Webhook handlers must deduplicate on event_id
applies_when:
  paths: ["**/webhooks/**/*.ts", "**/handlers/webhook*.ts"]
  symbols: ["onRequest", "POST", "handler"]
check:
  kind: ast-grep            # or "regex" | "shell" | "ts-module"
  pattern: |
    fastify.post($URL, async ($req, $reply) => {
      $$$BODY
    })
  must_contain:
    - "event_id"
    - "delivery_attempts"
    - "INSERT OR IGNORE"
severity: high
created: 2026-06-04
last_fired: 2026-06-09
fire_count: 7
fix_template: |
  // Check idempotency at the top of the handler:
  const existing = db.prepare(
    'SELECT id FROM delivery_attempts WHERE event_id = ?'
  ).get(req.body.event_id);
  if (existing) return reply.code(200).send();
---

# Webhook handlers must deduplicate on event_id

External webhook providers (Stripe, GitHub, etc.) retry delivery on failure.
Without idempotent handlers, retries process the same event multiple times —
which charges customers twice, sends two emails, or doubles a database update.

## When this applies
Any endpoint that accepts external webhook deliveries. Anything matching the
path glob in the frontmatter.

## What good looks like
The handler MUST:
1. Extract `event_id` from the verified signature payload (not from the body)
2. Check a persistent dedupe table (`delivery_attempts`) before processing
3. Return 200 OK on a duplicate (so the provider stops retrying)
4. Insert the dedupe row INSIDE the same transaction as the side effect

## Anti-patterns to call out
- Using an in-memory `Set` (lost on restart)
- Checking dedupe AFTER the side effect (race condition)
- Storing the entire body as the dedupe key (signatures rotate)

## Examples in this codebase
- `src/webhooks/stripe.ts:42` (good — uses INSERT OR IGNORE)
- `src/webhooks/legacy_zapier.ts:8` (bad — in-memory only, fix queued)
```

**Required fields:** `id`, prose body.
**Optional structured fields:** `domain`, `title`, `applies_when`, `check`, `severity`, `created`, `last_fired`, `fire_count`, `fix_template`.

**`check.kind` values (v1):**
- `regex` — simple text pattern. Cheap. **The only kind allowed in v1.0 alongside `ast-grep`.**
- `ast-grep` — ast-grep pattern. Fast, language-aware. Binary ships inside the container. **v1.5.**

**`check.kind` values DROPPED from v1 entirely (security):**
- ~~`shell`~~ — arbitrary shell execution. Cannot be safely sandboxed by volume-mount layout alone (the container has full network egress and reads project code rw via the corpus dir). v2 only, behind a real sandbox (`network_mode: none` + non-root + read-only rootfs for shell-rule exec, OR nested gVisor/firejail). Until that exists, no shell rules.
- ~~`ts-module`~~ — arbitrary TypeScript execution inside the container. Same security argument. v2 only, behind the same sandbox.

A rule with no `check` is **prose-only**. It still gets retrieved and injected; verification is the agent's responsibility. This is the dominant rule shape for v1 — most rules will be prose-only.

## Distributed Sub-Corpus Routing

`rules/_meta/domain-router.yaml` declares routing logic. The MCP server uses it to decide which sub-corpus(es) to query for a given context.

```yaml
routers:
  - id: by-path-glob
    rules:
      - if_match: "**/webhooks/**"
        domains: [idempotency, security, observability]
      - if_match: "**/auth/**"
        domains: [security, observability, naming]
      - if_match: "**/migrations/**"
        domains: [data-safety, observability]
      - if_match: "**/*.ts"
        domains: [maintainability, naming, perf]

  - id: by-intent-keyword
    rules:
      - if_intent_contains: ["test", "spec", "vitest"]
        domains: [testing, naming]
      - if_intent_contains: ["deploy", "rollout", "migration"]
        domains: [data-safety, observability, rollback]
      - if_intent_contains: ["agent", "harness", "judge"]
        domains: [agent-prompts, cost-control, robustness]

defaults:
  domains: [maintainability, naming]  # always include
  max_rules_per_domain: 4
  max_total_rules: 12
```

Each domain folder can have its own embedding index (auto-built on rule write), its own retrieval strategy, its own size discipline. Some domains stay small (10–20 rules, grep suffices). Some grow large (100+, semantic retrieval mandatory). The router decides.

## The Retrieval Service (`betterai-server`, inside container)

Single MCP server. **Stdio for local CLI agents that exec into the container; TCP (HTTP/SSE) for everyone else.**

### MCP tools exposed

```ts
// retrieve_rules(context, top_k=8) → Rule[]
{
  name: "retrieve_rules",
  description: "Get the most relevant rules for the current code context.",
  inputSchema: {
    context: {
      file_paths?: string[];         // current open files (host paths translated into mount paths)
      intent?: string;               // agent's stated plan or task
      symbols?: string[];            // function/class names in scope
      recent_diff?: string;          // last N lines of uncommitted diff
    };
    top_k?: number;                  // default 8
  }
}

// explain_rule(rule_id) → full rule body
// check_file(path | inline_contents) → { rule_id, line, evidence }[]
//   The VSCode extension passes inline file contents (not just path) to avoid
//   virtiofs/grpcfuse propagation lag on macOS Docker Desktop. CLI agents pass path.

// NOTE: log_application is server-internal, NOT an MCP tool exposed to agents.
// The server auto-logs to audit.jsonl on every retrieve_rules call and updates
// each returned rule's fire_count + last_fired. Agents cannot voluntarily log.
```

### Path translation (host ↔ container)

The MCP server expects paths in **project-relative form** (relative to the mounted projects root). Agents send absolute host paths; the server strips the `BETTERAI_PROJECTS_ROOT` prefix to get the project-relative path, then matches against `applies_when.paths` globs.

Per-platform host coordinate forms:
- **macOS / Linux:** `/Users/nicov/projects/foo/src/bar.ts` → strip `/Users/nicov/projects/` → `foo/src/bar.ts`
- **Windows-WSL:** `/mnt/c/Users/nicov/projects/foo/src/bar.ts` (WSL) or `C:\Users\nicov\projects\foo\src\bar.ts` (PowerShell) → install script must mount `/mnt/c/Users/nicov/projects` and normalize backslashes. Documented as Phase 1.5 work; Phase 1.0 is macOS-only.

Globs in `applies_when.paths` are matched against the project-relative path. `**/webhooks/**` matches `foo/src/webhooks/stripe.ts` correctly.

### Retrieval algorithm (v1)

1. Apply `domain-router.yaml` to the context → list of candidate domains.
2. For each domain:
   - If domain has an embedding index → top-k semantic retrieval against rule titles + bodies (MiniLM v2 inside the container)
   - Else → grep over `applies_when.symbols` + `applies_when.paths`
3. Merge results, rank by (domain-router priority × similarity × severity × inverse-age-since-last-fired).
4. Truncate to `max_total_rules`.
5. Return as structured list with `id`, `title`, `summary`, full body URL.

## The Docker Container

### Image

- Base: `node:22-bookworm-slim`
- Bakes in: `ast-grep` binary, `ripgrep`, `sqlite3` + `sqlite-vec` extension, `@xenova/transformers` model weights pre-cached, `@modelcontextprotocol/sdk`, `betterai-server` binary
- Image size target: <500 MB (model weights ~80 MB are the biggest single contributor)
- Multi-arch: `linux/amd64`, `linux/arm64` (macOS Apple Silicon, M-series)

### Default volume mounts

```yaml
# docker-compose.yml (the install path the user follows)
services:
  betterai:
    image: betterai/server:sha-${PINNED_SHA}      # SHA-pinned, NOT :latest (silent updates would change retrieval behavior)
    user: "${UID}:${GID}"                          # critical: container writes audit/embedding files as the host user
    ports:
      - "127.0.0.1:7777:7777"                     # MCP HTTP/SSE transport — localhost only, bearer-token-gated (see below)
    volumes:
      - ~/.betterai/rules:/data/rules:rw          # corpus (rw)
      - ~/.betterai/audit:/data/audit:rw          # audit log (rw, JSONL)
      - ~/.betterai/embeddings:/data/embeddings:rw # embedding cache (rw)
      - ~/.betterai/token:/data/token:ro          # bearer token, written by install script on first run
      - ~/projects:/projects:ro                   # user's code (read-only)
    environment:
      - BETTERAI_CORPUS_ROOT=/data/rules
      - BETTERAI_AUDIT_PATH=/data/audit/audit.jsonl
      - BETTERAI_PROJECTS_ROOT=/projects
      - BETTERAI_MCP_PORT=7777
      - BETTERAI_TOKEN_PATH=/data/token
      - BETTERAI_LOG_LEVEL=info
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "node", "-e", "fetch('http://localhost:7777/health').then(r=>r.ok?process.exit(0):process.exit(1))"]
      interval: 30s
      timeout: 3s
```

### MCP authentication (v1, mandatory)

`localhost:7777` binding alone is insufficient — any local process (browser tab via DNS rebinding, malicious `npm postinstall`, other dev tools) can hit the endpoint and read project code through `check_file`. v1 requires a **bearer token**:

- On first run, install script generates a random 32-byte token and writes `~/.betterai/token` (mode 0600).
- Container reads the token on startup; every MCP request must include `Authorization: Bearer <token>`.
- Server rejects requests where `Host` header is not `127.0.0.1:7777` or `localhost:7777` (DNS rebinding mitigation).
- The install script also prints the snippet to paste into Claude Code MCP settings, with the token interpolated.

### Container UID/GID (v1, mandatory)

Without `user: "${UID}:${GID}"` the container runs as root by default; the audit JSONL ends up root-owned on the host, and VSCode (running as the user) can't read it. The compose file MUST set `user` to the host UID/GID. Install script sources `$(id -u)` and `$(id -g)` and writes them into the compose file at install time (or uses `.env` interpolation).

### Image size + memory (honest budgets after review)

- **Image size budget: ~600 MB** (revised from <500 MB). Multi-stage build to strip dev deps. Slim base + MiniLM ONNX (~90 MB) + ast-grep (~50 MB) + ripgrep (~5 MB) + sqlite-vec extension (~5 MB) + node_modules (~150 MB) + server source (~5 MB) + seed rules (~1 MB) = ~410-470 MB realistic. 600 MB is the not-to-exceed.
- **Memory budget: ~1.5 GB steady state** (revised from <1 GB). MiniLM ONNX runtime + model weights resident is ~300 MB once a domain triggers embeddings. Lazy-load: don't load the model until a domain configured with embeddings is queried.

### Install / run

```bash
# Easiest: one-liner with a downloadable compose file
mkdir -p ~/.betterai/{rules,audit,embeddings}
curl -fsSL https://betterai.dev/install.sh | sh
# Above script:
#   1. Downloads docker-compose.yml to ~/.betterai/
#   2. Runs `docker compose up -d`
#   3. Runs `docker exec betterai betterai init --seed` (drops 30 starter rules)
#   4. Prints the MCP endpoint URL + the snippets to paste into Claude Code / claude.ai

# Alternative: plain docker run (no compose)
docker run -d --name betterai \
  -p 127.0.0.1:7777:7777 \
  -v ~/.betterai/rules:/data/rules \
  -v ~/.betterai/audit:/data/audit \
  -v ~/.betterai/embeddings:/data/embeddings \
  -v ~/projects:/projects:ro \
  betterai/server:latest
docker exec betterai betterai init --seed
```

### Path translation (host ↔ container)

The MCP server receives paths in **host coordinates** (`/Users/nicov/projects/foo/src/bar.ts`) from agents running on the host. It translates them to **container coordinates** (`/projects/foo/src/bar.ts`) before reading. Translation table is built from the volume mount config at startup.

### Cross-platform notes

- macOS Apple Silicon: works on Docker Desktop + Colima. arm64 image is the default.
- Linux: native Docker. amd64 or arm64.
- Windows: Docker Desktop on WSL2. Host paths are `/mnt/c/Users/...` inside WSL.
- The install script detects platform and adjusts the host paths in the compose file.

## The VSCode Extension (`betterai-vscode`)

Runs on the host. Talks to the container's MCP endpoint over TCP (`localhost:7777`).

### v1 surfaces

1. **Sidebar: Rules explorer** (TreeDataProvider)
   - Tree root: domains
   - Children: rules in that domain, sorted by `last_fired` desc (from container audit log)
   - Click rule: opens the markdown file in editor (file lives on host in `~/.betterai/rules/`)

2. **Status bar item: Corpus state**
   - `● 247 rules / 12 domains / last retrieval 3m ago`
   - Click → opens audit log webview
   - Health dot color: green (container up + responsive), yellow (degraded), red (container down)

3. **Webview: Audit log + retrieval inspector**
   - Tabular log of last 100 retrievals (read from `~/.betterai/audit/audit.jsonl`)
   - Click a retrieval row: full context + returned rule ids + Claude Code session id (if available)

4. **Commands (Command Palette)**
   - `BetterAI: Retrieve rules for current file` (manual test, returns top 8)
   - `BetterAI: Check current file against corpus` (calls `check_file()`)
   - `BetterAI: New rule from current selection` (scaffolds a markdown file with frontmatter)
   - `BetterAI: Open corpus root` (opens `~/.betterai/rules/`)
   - `BetterAI: Reindex embeddings` (runs `docker exec betterai betterai reindex`)
   - `BetterAI: Container status` (shows uptime, image version, mount info)
   - `BetterAI: Start/stop container`

5. **Diagnostics provider**
   - On file save, calls `check_file(path)` via the container MCP
   - Surfaces violations as VSCode diagnostics (squiggles + hover with rule title + severity)
   - Quick fix code action: "Apply fix from rule {id}" using `fix_template`

6. **Settings**
   - `betterai.mcpEndpoint` — default `http://127.0.0.1:7777/mcp`
   - `betterai.containerName` — default `betterai`
   - `betterai.corpusPath` — default `~/.betterai/rules`
   - `betterai.checkOnSave` — boolean (default: true)
   - `betterai.healthCheckIntervalSeconds` — default 30

### Container lifecycle in VSCode

- Extension activates → pings `localhost:7777/health`
- If container not running → status bar shows red dot + "Container offline (click to start)"
- Click → runs `docker compose -f ~/.betterai/docker-compose.yml up -d` via VSCode terminal
- All container interactions go through `docker exec betterai ...` for CLI ops; MCP calls go over TCP

All speak MCP. Same `localhost:7777/mcp` endpoint serves them all. Per-agent config is just an MCP URL.

### Optional Aether adapter (deferred)

If the user wants Aether's distributed agents to use the corpus with Aether's own per-role context budgets / batch retrieval, build a thin Aether-side wrapper around the same MCP calls. NOT in v1. Aether speaks MCP directly today.

## Distribution Plan

**v1:**
- **Container image:** `betterai/server:latest` published to Docker Hub + GHCR. Multi-arch (amd64 + arm64).
- **VSCode extension:** `.vsix` artifact in GitHub Releases. User installs via `code --install-extension betterai-*.vsix`. No marketplace presence yet.
- **Install script:** `curl -fsSL https://betterai.dev/install.sh | sh` — bootstraps `~/.betterai/`, fetches docker-compose, runs `docker compose up -d`, seeds 30 starter rules.
- **CLI:** `betterai-cli` (a small shim that exec's into the container) for things like `betterai-cli init`, `betterai-cli reindex`, `betterai-cli logs`.
- **CI/CD:** GitHub Actions build the image on push to `main`, tag with `latest` + commit SHA + semver.

**v2:**
- VSCode Marketplace + Open VSX publish
- Remote MCP transport hardening (auth tokens for non-localhost exposure)
- Corpus-share format (export/import a domain as a tarball)
- Self-learning hook: container logs "agent committed code that violates rule X" → promotes to user-review queue → user approves → rule sharpened

## Premises

1. **The corpus is the moat.** Not the framework. Not the IDE. Not the container. The rules.
2. **Docker container is the install vector.** `docker run` is the user's day-one experience.
3. **Agent-agnostic via MCP.** Any MCP-speaking agent works. Claude Code is the daily-driver.
4. **VSCode is the human surface.** Editor the user already lives in.
5. **Hybrid rule shape.** Prose mandatory; structured fields and executable checks optional.
6. **Distributed sub-corpora behind a single API.** Each domain scales independently; agents see one `retrieve_rules` call.
7. **Local-first, personal-toolkit.** No SaaS. No multi-user. No market polish. Robustness > polish.
8. **No code dependency on prior projects.** Aether, Aide, GBrain, techdebt — all separate. MCP clients, not code deps.

## Approaches Considered

### Approach A (CHOSEN) — Dockerized MCP server + VSCode extension + corpus

What this doc describes. Container is the install vector.
- **Effort:** ~4–6 weeks solo for v1.
- **Risk:** Low. Every piece (Docker, MCP, VSCode extension API, markdown corpus, sqlite-vec) is well-trodden.
- **Pros:** One-liner install. Sandboxed. Reproducible across platforms. Agent-agnostic.
- **Cons:** Adds Docker as a hard dep on the user's machine (most devs already have it).

### Approach B — MCP server only (no VSCode extension, no container)

Pure npm install. User runs `bun add -g @betterai/mcp`; Claude Code connects via stdio. No editor surface.
- **Effort:** 1–2 weeks solo.
- **Pros:** Ships in a week.
- **Cons:** No daily-driver win. Node version issues. Rejected by user constraint (Docker required).

### Approach C — Dockerized container + MCP server, no VSCode extension (deferred surface)

Ship the container in v1 (proves the corpus value); add the VSCode extension in v2.
- **Effort:** 2–3 weeks solo for v1.
- **Pros:** Fastest to a working corpus. Tests whether the corpus has signal before any editor code.
- **Cons:** No daily-driver visual surface in v1.

### Approach D — Same as A but add the eval lift harness (from v3 / techdebt design)

Bring forward the rate_limited_webhook_receiver fixture + 4-axis lift measurement. Validate the corpus actually lifts code quality.
- **Effort:** A + 2 weeks = ~6–8 weeks total.
- **Pros:** Hard evidence the corpus moves a number.
- **Cons:** Adds infra (worktree machinery, headless agent driver, autocannon). Delays daily-driver value.

## Audit JSONL Schema (v1, locked — designed for v2 self-learning)

Designing the audit log around v2's self-learning needs NOW prevents a painful schema migration later.

```jsonc
// One line per event. Append-only.
{
  "event_type": "retrieve" | "explain" | "check" | "rule_change" | "agent_apply",
  "ts": "2026-06-09T14:13:23.456Z",
  "agent_session_id": "claude-code:abc123",   // from MCP request metadata if available
  "tool_call_id": "tool_xyz",                 // MCP per-call id
  "context_hash": "sha256:...",               // hash of the context object (for retrieve_rules)
  "context": {                                // verbatim context sent
    "file_paths": ["..."],
    "intent": "...",
    "symbols": [...],
    "recent_diff": "..." | null
  },
  "router_decision": {
    "domains_queried": ["idempotency", "security"],
    "router_id_matched": "by-path-glob",
    "router_rule_index": 0
  },
  "rules_returned": [
    { "id": "...", "domain": "...", "score": 0.87, "reason": "embedding" | "grep" | "glob_only" }
  ],
  "latency_ms": 42,

  // v2 self-learning fields (populated post-hoc by a future analyzer; v1 leaves them null)
  "downstream_apply_event_id": null,    // set if the agent later applied a rule
  "downstream_commit_sha": null,        // set if we can correlate the retrieval to a git commit
  "downstream_violations": null         // set if check_file found violations the agent ignored
}
```

The MCP server writes one event per tool call to `~/.betterai/audit/audit.jsonl`. v2 will spawn a separate analyzer process that reads the JSONL and populates the `downstream_*` fields by correlating to git diffs in the projects mount.

## Open Questions

1. **Embedding model.** Default to `Xenova/all-MiniLM-L6-v2` (small, fast, baked into image)? Or use OpenAI embeddings via API (better quality, costs money)? Default: MiniLM v1; swap-out later. Container size budget assumes MiniLM.
2. **Path translation correctness across platforms.** Mounting `~/projects` differs on Windows-WSL. Install script handles it, but the translation logic needs cross-platform tests.
3. **MCP transport: HTTP vs stdio.** Claude Code supports both. HTTP simplifies multi-agent + container architecture; stdio is simpler for `betterai-cli`. Default: HTTP over `localhost:7777` for the container, stdio for the CLI shim.
4. **`check.kind=shell` and `ts-module` sandboxing.** Both execute arbitrary code inside the container. The volume mount layout (project dirs read-only, corpus rw, no host network access except 7777 inbound) is the sandbox. Document explicitly.
5. **Container auto-start on host boot.** macOS launchd vs systemd vs leave-it-to-user. Default: install script registers a systemd user unit on Linux, a launchd plist on macOS; Windows-WSL users manage manually.
6. **Rule file format vs gstack SKILL.md format.** Both are markdown + YAML frontmatter. Should BetterAI rules be importable from gstack skills (and vice versa)? Probably yes — gstack skills are a natural seed.
7. **Audit log retention.** Default 30 days, configurable. Older logs roll over to `~/.betterai/audit/archive/`.
