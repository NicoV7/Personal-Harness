# Design Extension: BetterAI v4.1 — Repo-Scoped Knowledge Bases

Generated by /plan-eng-review follow-up on 2026-06-09 (Wave 2 specialist-A).
Supersedes: ONLY the single-corpus assumption in v4 §"Distributed Sub-Corpus Routing" and §"The Retrieval Service". Everything else in v4 stands.
Status: LOCKED.

---

## Problem statement

The v4 design assumes a single corpus root at `~/.betterai/`. Every rule, skill, and memory the retrieval server can see lives under one host directory, owned by one user, mounted into the container at `/data/`. That model captures *personal* standards well — "I never let exceptions get silently swallowed", "I prefer dependency injection over service locators", "I always run the typechecker before committing" — but it has no story for *project-local* knowledge. A React codebase wants camelCase variable names even though I (globally) prefer snake_case. A repo using a particular Postgres extension wants "always pass `?sslmode=verify-full`" baked in. A repo that uses `pnpm` wants to forbid `npm install`. Today the only way to encode that is to author the rule globally and either (a) accept it bleeds into other repos, or (b) tag it with frontmatter and hope retrieval filters correctly. Both lose: (a) is wrong, (b) is fragile and requires a schema field that means "scope" but is really a path-derived property.

The user's directive on 2026-06-09 — "let's also support per-repo rules that ship with the code, with the repo version overriding the global version of the same id" — surfaces the gap at the right time. Phase 0 (corpus exists on disk) is done. Phase 1.0 (retrieval server scaffold) starts Day 3. If we lock scope semantics now, the Phase 1.0 server is built scope-aware from day one: dual-root walking, id-collision override, scope-tagged audit events. If we wait, we ship a single-corpus server, then migrate the audit-event schema, then re-test every retrieval path. The migration would not be hard, but it would be visible to consumers of audit logs and would require coordinated rollout. Catching it at design time costs nothing.

## The locked design

### 1. Two corpus roots

| Property      | GLOBAL                                  | REPO                                                      |
|---------------|-----------------------------------------|-----------------------------------------------------------|
| Host path     | `~/.betterai/{rules,skills,memories,_meta}/` | `<repo-root>/.betterai/{rules,skills,memories}/`     |
| Container path| `/data/{rules,skills,memories,_meta}/`  | `/projects/<repo-rel>/.betterai/{rules,skills,memories}/` |
| Mount         | dedicated `~/.betterai:/data:rw`        | piggybacks on existing `~/projects:/projects:ro`          |
| Source of truth | baked into image at `/opt/betterai/seed-corpus/`, copied on first run | version-controlled with the repo, ships with the code |
| Authoring     | VSCode "New rule" or `betterai new rule --scope global` | `betterai new rule --scope repo` (default when CWD is inside a git repo) or hand-write |
| `_meta/`      | YES — conflict-resolution, router config, etc. | NO — the repo corpus inherits meta from global         |

Same frontmatter schema. Same directory shape inside both roots. Same body sections. **Scope is implicit from which root the file lives in** — there is no `scope:` field in frontmatter. A rule "is repo-scoped" iff its host path starts with `<repo-root>/.betterai/`; otherwise it is global. This is the single most important property of the design: the schema does not need a migration, and a file moved from one root to the other automatically changes scope.

### 2. Detection algorithm

Server-side, on every `retrieve_context` (and every `retrieve_rules` / `retrieve_skills` / `retrieve_memories` / `record_memory`) call:

```
Step 1: take context.file_paths[0] (or context.repo_root if explicitly provided).
Step 2: translate from container path to host path (existing v4 logic, §Path translation).
Step 3: walk up parent directories of the host path until you hit a directory
        containing a .git/ entry. That host directory is the repo root.
Step 4: check if <repo-root>/.betterai/ exists AS A DIRECTORY (not a regular file,
        not a symlink to nowhere). If yes, that is the repo corpus.
Step 5: cache the (host_path_prefix -> repo_root) mapping for 60 seconds, keyed
        by the mtime of <repo-root>/.git/HEAD so the cache invalidates on branch
        switch or HEAD update.
```

If `context.file_paths` is empty AND `context.repo_root` is not provided, skip steps 1–4 and query only the global corpus. The agent's caller is responsible for providing at least one path when it wants repo-scoped knowledge.

The cache key is the host path prefix (e.g. `/Users/nicov/work/myapp/src/...` resolves to repo root `/Users/nicov/work/myapp`); the cache value is the repo root plus the boolean "has `.betterai/` subdir". A 60-second TTL gives a forgiving window for the common case (user edits within one repo for an hour) without making branch switches feel stale. `git checkout other-branch` rewrites `.git/HEAD`'s mtime, which invalidates the entry.

### 3. Merge and conflict resolution

1. Run the domain-router and grep retrieval **independently against both corpora**. Each produces its own candidate set.
2. Build a merged candidate set.
3. **ID-collision rule**: if a rule id appears in both corpora, the **repo version replaces** the global version in the response. The global is dropped — the agent never sees it. This is the "extends/overrides" pattern: the repo file is the whole truth, not a frontmatter merge.
4. **Non-collision additive**: repo-specific items with unique ids ADD to the global set. Both appear, ranked normally.
5. **Ranking**: severity × match-strength × recency, as v4. Repo gets **no automatic boost** — it only wins at id-collision.
6. The override is recorded in the audit event (see §6 below).

The mental model: the global corpus is the baseline, the repo corpus is a patch. A patch can add files (additive) or replace files (override). It cannot partially-modify a file, because there is no frontmatter merge.

### 4. Conflict priority — extends `_meta/conflict-resolution.md`

The existing kind-level priority (from team-η's first draft) stays as the outermost tiebreaker:

```
memory(kind=decision, durability=long)  >  rule  >  skill  >  other memories
```

NEW within-kind tiebreaker — added to the same document:

```
At id-collision (same id in both global and repo):
    REPO version wins. The global version is DROPPED from the response,
    not merged.

At distinct ids (no collision, both items happen to match the same query):
    Both items appear in the response; ranking decides ordering.
```

**Worked example.** Global rule `use-snake-case` lives at:

```
~/.betterai/rules/STANDARDS/naming/use-snake-case.md
  severity: medium
  body: "Use snake_case for variable, function, and file names."
```

I now start work on a React project. The repo authors:

```
<repo>/.betterai/rules/STANDARDS/naming/use-snake-case.md
  severity: high
  body: "Use camelCase for variable names per React convention.
         File names stay kebab-case. Underscores are reserved for
         internal-only helpers."
```

Both files have id `use-snake-case` (the id is derived from the path under `rules/`, which is the same in both corpora).

- Agent working in that React repo: the retrieve_context response contains **only the repo version**. The audit log records `overridden_global_ids: ["use-snake-case"]`.
- Agent working in a non-React repo (no override file): the response contains **only the global version**.
- Agent with no repo context at all (e.g. an ad-hoc prompt with no `file_paths`): the response contains only the global version.

This is the desired behavior: the project's "different in this codebase" intent is expressed by re-declaring the rule with the same id, and the override is visible in the audit log.

### 5. MCP tool surface changes

```
retrieve_context(context, top_k_per_kind, scope?)
  scope: "merged" | "global" | "repo"   (default: "merged")
  Response shape: every returned rule/skill/memory gains a 'scope' field
                  with value "global" or "repo".

retrieve_rules(query, scope?)
retrieve_skills(query, scope?)
retrieve_memories(query, scope?)
  Same scope option, same default ("merged").

record_memory(memory, scope?)
  scope default: "repo" if a repo root is detected in the current call's
                 context, else "global".
  Server writes to:
    ~/.betterai/memories/<yyyy-mm>/<id>.md          if scope=="global"
    <repo-root>/.betterai/memories/<yyyy-mm>/<id>.md if scope=="repo"
```

Explicit `scope` overrides the default in every tool. `scope="repo"` when no repo root is detectable is a validation error (returned as a structured error, not an exception). `scope="global"` always works.

The `scope` field on returned items is required and is the only new field. Existing consumers ignore unknown fields, so the change is backward-compatible.

### 6. Audit log schema additions

Add the following fields to every `retrieve_*` and `record_memory` audit event:

| Field                       | Type                                    | Meaning                                                                          |
|-----------------------------|-----------------------------------------|----------------------------------------------------------------------------------|
| `repo_root_detected`        | `string \| null`                        | Host path of the detected repo root; `null` if none.                             |
| `scopes_queried`            | `["global"] \| ["repo"] \| ["global","repo"]` | Which corpora the retrieval actually ran against.                          |
| `rules_returned[].scope`    | `"global" \| "repo"`                    | Per-returned-item scope. Also added to `skills_returned[]` and `memories_returned[]`. |
| `overridden_global_ids`     | `string[]`                              | Global ids dropped because a repo version replaced them. Diagnostics only.       |

These fields are always present (use empty array / null when N/A) so log consumers can rely on the shape.

### 7. CLI additions

```
betterai new rule [--scope repo|global]
    Default: repo if (CWD is inside a git repo) AND
                    ((<repo-root>/.betterai/ exists) OR
                     (user has not explicitly opted out via betterai config));
             else global.
    If repo selected and <repo-root>/.betterai/ does not exist:
        scaffold it — mkdir -p the four subdirs (rules, skills, memories)
        and write a one-line README.md explaining "this directory is a
        BetterAI per-repo corpus; see https://betterai/docs/repo-scoping".
```

```
betterai status
    GLOBAL  ~/.betterai/                  13 rules / 5 skills / 5 memories
    REPO    /Users/nicov/BetterAI/.betterai/  5 rules / 0 skills / 0 memories
    OVERRIDES (last retrieve): 1 global rule overridden by repo rule "mcp-tools-require-bearer"

    The OVERRIDES line is shown only when the audit log's last retrieve
    event has a non-empty overridden_global_ids array. Otherwise omitted.
```

```
betterai why <task> --context <file>
    Annotate each returned item with [GLOBAL] or [REPO] in the output.
    Items that overrode a global version are annotated [REPO; overrides global].
```

```
betterai validate
    Validate BOTH corpora. Id-collisions are reported as INFO, not error
    — they are intentional overrides, not duplicates. Other validation
    rules (broken refs, missing required frontmatter fields, malformed
    severity) remain errors as before.
```

### 8. What does not change

- The rule/skill/memory frontmatter schemas (corpus-eng-review §1.2 + multi-agent-eng-review §1.2/§1.3).
- The retrieval algorithm (router → grep → rank), except for being run twice and merged.
- The bearer-token auth, the UID/GID model, the Docker mount layout — the existing `~/projects:/projects:ro` mount already gives the server read access to every `<repo>/.betterai/` it needs.
- The Phase 1.0 budget — this is a Day-3 server work item, not a new phase.

### 9. Why this design

- **Version-controlled per-repo knowledge.** A project's "use camelCase here" rule ships with the project, gets PR'd like code, survives team handoffs, and is reviewed by humans before it lands.
- **Personal global rules stay global.** "No catch-all exception masking" applies everywhere — it's a property of me, not of any one project.
- **Override pattern is the natural shape.** A repo says "this rule is different HERE" by re-declaring with the same id. That's the same gesture as a derived class overriding a method.
- **No schema change required.** Scope is a property of *where the file lives*, not *what it says*. Moving a file between roots reclassifies it instantly.
- **The audit-log delta makes overrides visible.** Future-you (or future-coworker) can see "the repo overrode rule X" without surprise, by reading the audit event.

| Codepath | Failure | Rescued? | Test? | User sees? | Logged? | Critical gap? |
|----------|---------|----------|-------|------------|---------|---------------|

| `<repo>/.betterai/` exists but is a regular file (not a directory) | `os.path.isdir` returns false; we treat as "no repo corpus" | YES — fall back to global-only | YES — `test_repo_corpus_path_is_file` | One-time warning on startup of the file watcher: "`.betterai` at `<path>` is a file, not a directory; ignoring." | YES — at warn level | No |
| `<repo>/.betterai/` is a symlink to a non-existent target | `os.path.isdir` returns false via dangling symlink | YES — fall back to global-only | YES — `test_repo_corpus_dangling_symlink` | Same warning as above | YES | No |
| Id collision across repo and global (intentional override) | Not a failure — by design | N/A | YES — `test_id_collision_override` and `test_id_collision_audit_log` | Yes — `betterai status` "OVERRIDES" line and `betterai why` `[REPO; overrides global]` annotation | YES — `overridden_global_ids` | No |
| `.git/HEAD` mtime cache stale across branch switches | User switches branch, cached repo-root entry shows old `.betterai/` state until 60s expires | YES — cache key includes `mtime(.git/HEAD)`; `git checkout` rewrites HEAD; mtime change invalidates the entry on next access | YES — `test_branch_switch_invalidates_cache` | No, transparent | NO — at debug level only | No |
| `record_memory` called with no detectable repo root | Default scope would be "repo", but we can't write anywhere | YES — fall back to `scope="global"` with a warning in the response payload (`scope_fallback: "global"`) | YES — `test_record_memory_falls_back_to_global` | Yes, in the response payload | YES — `scope_fallback: "global"` field on the audit event | No |
| Container cannot read `<repo>/.betterai/` because the projects mount excludes the repo | Bind mount is `~/projects:/projects:ro`; a repo at `~/code/foo` is invisible to the container | YES — repo-root detection succeeds (we read host paths), but the `os.path.isdir` check fails inside the container; we fall back to global with a structured warning | YES — `test_repo_outside_projects_mount` | Yes — `betterai status` shows a warning: "repo detected at `<host>` but not reachable in container; corpus skipped" | YES — `repo_corpus_unreachable: true` field on the audit event | NO — but documented as a config issue (user adds another mount or moves the repo) |
| VSCode "New rule" command in a repo without `.betterai/` | User invokes the command; we need to decide scaffold-or-warn | YES — scaffold the four subdirs and the README.md, then proceed. Always show a one-line toast: "Scaffolded `<repo>/.betterai/` — commit this directory to share repo-scoped knowledge." | YES — `test_new_rule_scaffolds_repo_corpus` and `test_new_rule_toast_shown` | Yes, toast in VSCode | YES — `corpus_scaffolded: true` event | No |
| Two retrieve_context calls concurrent on the same repo, cache miss-and-fill race | Both calls walk up, both check `.betterai/`, both write the cache entry; benign | YES — last write wins; both entries are identical so the race is harmless | YES — `test_concurrent_cache_fill` | No | NO — debug only | No |
| File added to `<repo>/.betterai/rules/` while server is running | Server's file watcher must pick it up | YES — file watcher already covers `/data/` for v4; extend to walk every detected repo corpus | YES — `test_repo_corpus_hot_reload` | Yes via `betterai status` reflecting new counts | YES — `corpus_reload: {scope, path, file}` | No |
| `<repo>/.betterai/` deleted at runtime | File watcher sees the deletion; cached `(prefix, repo_root)` still points to a dead corpus | YES — file watcher invalidates the cache entry on directory delete; next retrieve walks again and finds no `.betterai/` | YES — `test_repo_corpus_deleted_at_runtime` | Yes via `betterai status` | YES | No |

**Critical gaps: 0.** Every failure mode is rescued, tested, logged, and (where the user benefits from knowing) surfaced.

## Unresolved decisions

1. **Should a repo corpus be allowed to have its own `_meta/`?** Auto-decide: **NO** for v4.1. The repo inherits all `_meta/` from global. A repo that wants to override the conflict-resolution policy is a rare and dangerous case; we can add it in v4.2 if a real need surfaces. Revisit if any team requests it during Phase 1.1.
2. **Should we support nested repos (a `.git/` inside another `.git/`)?** Auto-decide: **YES, inner wins.** The walk-up stops at the first `.git/`. If a submodule lives at `<repo>/vendor/foo` and has its own `.betterai/`, an agent editing inside `vendor/foo` sees that corpus, not the parent's. Submodules are independent projects with independent rules. Test: `test_nested_repo_inner_wins`.
3. **Should `.betterai/` be gitignored or version-controlled?** Auto-decide: **version-controlled by default.** The whole point is that repo-scoped knowledge ships with the code. The README.md scaffolded by `betterai new rule` will say "this directory is intended to be committed; do not gitignore it." A team that wants per-developer-local rules can `.git/info/exclude` individual files, but the default is "in the repo."

## What I noticed about how you think

You surfaced the scoping question at the exact right moment in the project lifecycle. Phase 0 (the corpus exists on disk, schema is locked) just finished. Phase 1.0 (the retrieval server scaffold) starts in two days. If the question had come a week later, we would have built a single-corpus server, then had to migrate the audit-event schema and re-test every retrieval path. The migration would not be hard, but it would be visible — log consumers (the dashboards, the `betterai why` CLI, the multi-agent eval harness) would all need coordinated updates. By catching the scope dimension at design time, the Phase 1.0 server is built scope-aware from day one and pays zero migration cost.

What this looks like as a pattern: you let the foundational work settle before adding the next dimension, then you add it as a delta against a stable base rather than negotiating it into a moving target. The corpus design didn't fork to chase the scoping question; the scoping question got its own extension doc that locks one decision and stops. That keeps the v4 design readable as the foundational artifact and this extension readable as the single-purpose delta — which is exactly the shape future-you will want when re-reading these in three months to remember why repo corpora work the way they do.
