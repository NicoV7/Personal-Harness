## Section 1: Architecture Review — Multi-Agent Consumption + Skills/Memories

### 1.1 The Three Artifact Types (Locked Schema)

Today's v4 corpus has only `rules/`. This review adds two siblings:

| Artifact | Purpose | Retrieved when | Authored by | Mutates over time |
|---|---|---|---|---|
| **Rule** | Hard constraint or anti-pattern. Read BEFORE writing code. | Agent about to plan / write / refactor / review code. | Human (you), via VSCode "New rule" command. | Slow — bumps to severity only. |
| **Skill** | Procedure / how-to. Read when about to PERFORM a recurring task. | Agent's intent matches a procedure (e.g., "add a new MCP tool", "write a Vitest test"). | Human OR codified from a successful flow (`/skillify` pattern). | Medium — procedures get sharpened. |
| **Memory** | Cross-session learning. "I tried X and Y happened." | Agent planning / debugging matches a prior episode by context. | Auto-captured (audit-log post-processor) OR human via `record_memory` MCP tool. | Fast — written every session. |

Rules are constraints (NOT this); skills are procedures (DO this); memories are episodes (LAST time we tried this, X happened). The three are retrieved by separate calls and injected into the agent's context in distinct sections.

### 1.4 MCP Tool Surface (REVISED — v4 + Multi-Agent)

v4 design exposes 3 tools: `retrieve_rules`, `explain_rule`, `check_file`. This review adds:

```ts
// EXISTING (unchanged signatures):
retrieve_rules(context, top_k=8) → Rule[]
explain_rule(rule_id) → Rule body
check_file(path | inline) → Violation[]

// NEW:
retrieve_skills(context, top_k=4) → Skill[]
  // Same context shape as retrieve_rules. Returns procedural how-tos
  // for what the agent intends to do.

retrieve_memories(context, top_k=4) → Memory[]
  // Returns relevant prior episodes. Includes "Don't relitigate" sections
  // for any decision-kind memories matching the context.

retrieve_context(context, top_k_per_kind=4) → {rules: [], skills: [], memories: []}
  // Convenience aggregator. Single call from a subagent → all three lists.
  // This is the recommended call for subagents (cheaper than 3 calls;
  // single audit event; consistent context_hash).

record_memory(memory) → { id }
  // The agent can record a memory after a discovery / decision / failure.
  // Server validates frontmatter, writes to memories/<yyyy-mm>/<id>.md.
  // Authoring rules and skills stays human-driven (VSCode "New rule");
  // memories are agent-writable because episodes happen during agent work.
```

`retrieve_context` is the **canonical subagent entry point** — one call, one audit event, three lists. The other granular tools remain for direct human use (VSCode commands) and for fine-grained agent retrieval when needed.

### 1.5 Multi-Agent Propagation — How Subagents Reach BetterAI

There are five subagent classes in the current environment. Each has different MCP propagation behavior:

| Class | How spawned | MCP visibility | BetterAI auto-reach? |
|---|---|---|---|
| **A. Main-loop Claude Code** | User invokes `claude` | All configured MCP servers | YES (direct) |
| **B. `Agent` tool subagent** | Main loop calls `Agent({subagent_type, prompt})` | Inherits parent's MCP server set; tool schemas resolved via ToolSearch in the subagent's context | YES, if subagent's `Tools` allowlist includes MCP tools (default for `general-purpose`, `claude`) |
| **C. Workflow `agent()` call** | `Workflow` runs script with `agent(prompt)` | Per-agent ToolSearch; schemas load on demand | YES, but requires the subagent to ToolSearch `"betterai"` to load schemas before calling |
| **D. Background agent** | `Agent({run_in_background: true})` | Same as B | YES (same as B) |
| **E. Scheduled / cron** | CronCreate / RemoteTrigger | Per-tool registry; interactively-auth'd MCP servers MAY be absent | **PARTIAL — known limit:** in headless/cron runs, MCP servers requiring interactive OAuth may not be reachable. BetterAI's bearer-token auth (no OAuth) makes it reachable as long as the container is up. Document this. |

**Critical implication:** v4 design says "stdio for local CLI agents that exec into the container; TCP (HTTP/SSE) for everyone else." For multi-agent consumption, **stdio is wrong for everyone except the main-loop CLI shim.** Subagents and workflow `agent()` calls do NOT fork the main loop's stdio. They make new MCP connections. Only HTTP/SSE works across the subagent boundary.

→ **v4 should change to:** HTTP/SSE is the default and only transport for `retrieve_rules` / `retrieve_skills` / `retrieve_memories` / `record_memory`. The stdio path is dropped from v1 entirely. The `betterai-cli` shim already uses `docker exec` for housekeeping ops; it doesn't need MCP-over-stdio.

### 1.6 Auto-Retrieval Mechanism (the load-bearing question)

The user said *"Each agent should search the brain for skills and memories to use automatically."* Today, retrieval is voluntary — the agent decides to call `retrieve_rules` if it thinks it needs to. Subagents (especially `Workflow` `agent()` calls) routinely don't think to.

**Three levers considered:**

| Lever | What it does | Cost | Reliability for subagents |
|---|---|---|---|
| **a. System prompt directive** | Append "Before writing >10 LOC or proposing architecture, call retrieve_context with your current task" to every Claude Code session's system prompt when BetterAI MCP is configured | Free; one-time install step | **HIGH for main loop. LOW for subagents** — the orchestrator writes subagent prompts and may forget to include the directive. |
| **b. Tool-call interceptor / sentinel** | A "first-write-tool" sentinel: the MCP server records when the agent calls an MCP tool that writes code; if no `retrieve_*` was called in the last 60s with a matching context_hash, return a soft warning in the response and log a "skipped-retrieval" event | Server-side only | **MEDIUM** — only catches after-the-fact; doesn't prevent the write. |
| **c. Skill-style auto-trigger** | Treat BetterAI as an auto-triggering skill (like the user's gstack auto-trigger skills): the description field on a `betterai` skill says "ALWAYS invoke before substantive code/plan/refactor". The agent's harness loads this as part of the active skill set; it fires before the agent's first write. | Per-host install (skill file in `~/.claude/skills/`) | **HIGH for main loop and Claude-Code-spawned subagents.** **UNKNOWN for `Workflow` `agent()` calls** — workflow subagents may not load the user's skill set. |

**RECOMMENDATION: belt-and-suspenders.**

- **Lever (a) is mandatory** — the install script appends a CLAUDE.md preamble OR sets `BETTERAI_SYSTEM_PROMPT_PREFIX` so every session starts with the retrieval directive.
- **Lever (b) is recommended** — cheap server-side audit signal that catches misses; surfaces in a Phase 2 VSCode "missed retrieval" panel.
- **Lever (c) is OPTIONAL** — install script installs a `~/.claude/skills/betterai-auto-retrieve/SKILL.md` if the user opts in. Workflow-internal agents fall back to lever (a) + (b).

This is the **honest answer:** there's no MCP-native "ALWAYS call this tool first" primitive. The user's quoted intent ("automatically") is achievable for the main loop and Claude-Code-spawned subagents via levers (a)+(c). Workflow `agent()` calls require the orchestrator (the script's prompt strings) to include the retrieval instruction. That responsibility is documented, not enforced.

## Section 4: Performance Review

### 4.1 At v1 scale (13 rules, ~10 skills, <50 memories) — multi-agent

| Concern | 1 subagent | 5 parallel subagents | 50 parallel subagents |
|---|---|---|---|
| `retrieve_context` p50 latency | <20ms | <50ms (cache hits for shared context) | <200ms (queue depth) |
| `retrieve_context` p99 latency | <50ms | <100ms | <500ms (queue + 429 rejects above 16) |
| Server CPU resident | ~5% | ~15% | ~80% (queue serialized) |
| Server RSS | ~400 MB | ~500 MB (cache resident) | ~600 MB |
| Audit JSONL write throughput | <1 KB/s | <5 KB/s | <50 KB/s peak |
| Cache hit rate | 0% (single call) | ~60% (same-context fan-out) | ~85% (highly correlated fan-out) |

The 16-concurrent limit + queue means a 50-agent fan-out has tail latency in the half-second range. Acceptable for v1; revisit if a real workflow needs >20 concurrent retrievals.

### 4.2 At v1.5 scale (50+ rules, 30+ skills, 200+ memories) with embeddings on

- MiniLM single-threaded inference is the bottleneck. At 16 concurrent retrieves with no cache, latency dominates. Mitigations:
  - context_hash cache (already specced)
  - "skim mode" path: if `retrieve_context` is called with `top_k_per_kind=1`, skip embedding for low-priority domains and use grep-only — cheaper fast path.
- Memory directory growth: at 12 memories/month average, after 1 year that's 144 files. Date-sharded by yyyy-mm keeps any single directory <50 files. No problem.

### 4.3 N+1 Check

- `retrieve_context` does a single domain-router pass + per-kind retrieval. No N+1.
- `record_memory` is a single fs write + single audit append. No N+1.
- Audit webview (Phase 2) reading the JSONL streams a single file. No N+1.

### 4.4 Caching opportunities (locked for v1)

1. **context_hash → result** LRU, 256 entries, 60s TTL — specified.
2. **rule/skill frontmatter parsed cache**, keyed by file mtime — deferred to Phase 1.5 (file count is small in v1).
3. **embedding vectors per rule** — already in sqlite-vec when embeddings turn on (Phase 1.5).
