## Step 0: DX Investigation

### 0A. Developer Persona (auto-detected; explicit framing)

BetterAI is a *personal toolkit, not a product* (user's explicit framing across the v4 design). DX evaluation must respect that bar:

**PRIMARY PERSONA — "Solo author + dogfooder" (Nico, you):**
- Context: building this for yourself; you install it on your own machines; you write every seed rule by hand; you measure the lift on your own work.
- Tolerance: very high for short-term friction (you'll debug your own container), *very low* for friction in the daily-driver loop. If `retrieve_rules` adds 500ms to every Claude Code call, you'll uninstall.
- Expects: the moat is the corpus, so corpus authoring must be the lowest-friction surface. Container ops are 1-time pain. VSCode browsing is occasional.
- Abandon trigger: if a single day of "author 13 rules" feels like 4 days of yak-shaving on schema validators, the project dies in Phase 0.

**SECONDARY PERSONA — "Other AI tinkerer who learns about BetterAI" (someone hearing about it in 2027):**
- Context: hears you talk about it, wants to try it. Not your customer. Not entitled to polish.
- Tolerance: ~10 minutes to first signal. If they have to git-clone + read 3 docs + author rules themselves to evaluate, they bounce.
- Expects: a seed corpus that demonstrates the lift on day one. A way to consume your seed corpus without committing to maintaining one.
- Currently *out of scope* per "personal toolkit" framing — but the seed corpus is the artifact that ports across this boundary for free, so optimize for them as a side effect.

**TERTIARY "PERSONA" — the AI agent itself (Claude Code, subagents, claude.ai):**
- Context: receives `retrieve_rules` results mid-prompt; must parse and apply them.
- Tolerance: zero. Every wasted token of retrieved-but-irrelevant rules is context budget burned.
- Expects: rules that are short, agent-friendly (declarative, not narrative), and have unambiguous "what good looks like" sections it can pattern-match against.
- Abandon trigger: low-quality retrieval (rules that don't match the context) trains the agent (and you) to ignore the retrieval signal.

**AUTO-DECIDE:** primary persona = you (solo author + dogfooder). All scoring below is calibrated to that bar. Secondary persona considered only where the work overlaps (e.g., README is "free" optimization for both). Tertiary persona (the agent) is the consumer side of every scoring dimension — the rule body schema IS its API.

### Pass 1: Getting Started Experience — **4/10** (target: 8/10)

**Evidence:** TTHW measured at 5-12 min (0C/0G), target tier Competitive (2-5 min). Voluntary retrieval (issue from 0F/0G) is the biggest gap.

**What a 10 looks like for BetterAI:** `curl install.sh | sh` runs in <60s, drops the seed corpus, writes the MCP config snippet directly into `~/.claude/mcp_settings.json`, prints a welcome-task command. User opens Claude Code, pastes the welcome prompt, sees retrieve_rules fire in audit log, sees Claude write differently. T+3:00 = first magical moment.

**Gap-to-10 fixes (DX-FIX-1 through DX-FIX-4):**

- **DX-FIX-1 [CRITICAL]:** Add `~/.betterai/welcome-task/` seeded by `betterai init --seed`. Single README with a copy-paste prompt that's *known* to fire ≥3 seed rules. Documented success: "tail the audit log; you should see three retrievals in <60s."

Even a minimal `install.sh` that just (a) writes docker-compose.yml to `~/.betterai/`, (b) runs `docker compose up -d`, (c) appends an MCP block to `~/.claude/mcp_settings.json` with the bearer token interpolated, (d) prints the welcome-task command. Cuts steps 1-5 to a single command.
- **DX-FIX-3 [HIGH]:** Make auto-retrieve lever (a) — system-prompt prefix — part of the install script. Without this, TTHW is bounded by the agent's whim. (Cross-references prior eng review's TD-M7.)
- **DX-FIX-4 [HIGH]:** Add a `betterai status` CLI command that prints: container up, last 5 retrievals, last 5 audit events, current corpus size. Replaces "tail the JSONL and squint" with a single readable command. Confidence and discoverability win.

**Re-score after DX-FIX-1..4:** 8/10 — competitive tier, magical moment delivered in <3 min on a fresh laptop.

**What a 10 looks like:**

- CLI: `betterai init / validate / status / replay / why <task>` — 5 verbs, each does one thing.
- Rule authoring API: `betterai new rule --category STANDARDS --domain maintainability --id simplicity-first` scaffolds a complete .md file with all required H2 sections pre-stubbed. Author fills prose; never types frontmatter.

**Gap-to-10 fixes:**

- **DX-FIX-5 [HIGH]:** Spec the `betterai-cli` v1 commands now. Minimum 5: `init` (seed/reset corpus), `validate` (schema-check all rules), `status` (printable health + recent activity), `why <rule-id> --context <ctx>` (explain why a rule did/didn't retrieve), `replay --since 7d` (audit log digest). Each `--help` is a usable 1-screen reference.
- **DX-FIX-6 [HIGH]:** Add a `betterai new rule` / `betterai new skill` / `betterai new memory` scaffolder. Pre-stubs frontmatter (required fields with placeholders), pre-stubs all H2 sections with prompts ("describe the anti-pattern here"). Author writes prose; validator catches missed fields. This is *the* DX leverage for the rule-authoring workflow that 0F flagged.
- **DX-FIX-7 [MEDIUM]:** Default `retrieve_context` to the recommended entry point in all docs and the README; demote the 3 granular tools to "advanced." Single canonical call > three competing ones.
- **DX-FIX-8 [MEDIUM]:** Add an MCP tool description (the `description:` field that the agent reads when discovering tools) that says: "ALWAYS call retrieve_context as your first action on any code task. Returns the rules, skills, and prior episodes relevant to your current intent." This is the lightweight version of auto-retrieve lever (a) that works for *any* MCP-speaking agent without an external system prompt.

**Re-score:** 8/10. Author scaffolder + 5-verb CLI + canonical MCP entry point.

### Pass 3: Error Messages & Debugging — **3/10** (target: 8/10)

**Evidence:** Not addressed anywhere in v4 design or prior eng reviews. Three error paths to trace:

1. **Container can't reach corpus mount.** Today: container fails to start. User sees `docker logs betterai` → some Node stack trace about ENOENT on `/data/rules`. *No fix in the message.*
2. **Bearer token mismatch.** Today: every MCP call returns 401. Claude Code shows "MCP tool error" — opaque. *No diagnostic.*
3. **Rule file has malformed frontmatter.** Today (per prior eng review's validator spec): server logs an error and skips the rule. *User never sees this unless they tail the container log.*

**What a 10 looks like — three-tier error structure (per Hall of Fame: Elm/Rust/Stripe):**

```
ERROR: corpus directory not found at /data/rules
  Cause: the host directory ~/.betterai/rules does not exist or is not mounted.
  Fix:   run `mkdir -p ~/.betterai/rules && docker compose up -d`.
  Docs:  https://github.com/you/betterai#corpus-mount
```

**Gap-to-10 fixes:**

- **DX-FIX-9 [CRITICAL]:** Spec a top-level error taxonomy: container startup errors, MCP request errors, schema validation errors, retrieval miss diagnostics. Each error class gets a `[BAI-NNN]` code linked to a one-page doc. Use Stripe's API error shape for MCP-returned errors: `{type, code, message, hint, doc_url}`.
- **DX-FIX-10 [HIGH]:** `betterai validate` (DX-FIX-5) emits the same three-tier error per malformed rule: what's wrong, why it matters, exact fix. Filename + line number for every error.
- **DX-FIX-11 [HIGH]:** Add a `betterai why <task-description>` debug command that simulates a `retrieve_rules` call against a hand-written context and explains the routing decision step-by-step: "(1) domain-router matched `by-path-glob` → domains [security, idempotency]; (2) grep in security/* found 2 matches; (3) ranking: rule X scored 0.87 (severity high × match 1.0 × recency 0.87)". Removes the "why didn't retrieval fire?" black box.

**Re-score:** 8/10 with all three fixes. Error messages with a fix are 10× more valuable than "this failed."
