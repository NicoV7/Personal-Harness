## Section 1: Architecture Review — Standardized Rule Schema + Two-Axis Taxonomy

### 1.1 The Standardized Rule File

**File location:** `rules/<category>/<domain>/<id>.md`

Two-axis taxonomy:
- **Category (techdebt-aligned, ~5 buckets, slow-growing):** what KIND of rule is this?
- **Domain (v4-aligned, growing freely):** what CONCERN does this rule address?

Why two axes: techdebt's 5-doc structure (STANDARDS, PROCESS, PATTERNS, ARCHITECTURE, DOCUMENTATION) describes the *register* of the rule (constraint vs workflow vs canonical pattern). v4's domain (security, perf, idempotency, …) describes the *concern*. A rule has both: "no catch-all exception handlers" is a STANDARDS-category rule in the maintainability/error-handling domain. The agent retrieves by domain (context-relevant); the human browses by category (mental organization).

**Five categories (taxonomy locked):**

| Category | Meaning | techdebt analogue | Examples from RULES.md |
|---|---|---|---|
| `STANDARDS` | Hard constraints the agent MUST follow | STANDARDS.md | "Simplicity First", "Avoid Overly Defensive Programming" |
| `PROCESS` | Workflow / how the agent should approach a task | (techdebt has these implicit in hooks) | "Search brain BEFORE substantive work", "Think Before Coding", "Goal-Driven Execution", "Ask about requirements" |
| `PATTERNS` | Canonical implementations to reuse, not reinvent | PATTERNS.md | (none in current RULES.md; future rules like "use this dedupe helper not your own") |
| `ARCHITECTURE` | Structural / system-shape rules | ARCHITECTURE.md | "Layer architecture in service/models/dtos/tests/tools/constants/handlers", "Avoid GOD files >2000 LOC" |
| `DOCUMENTATION` | Rules about doc/comment behavior | DOCUMENTATION.md | (none in current RULES.md) |

`PROCESS` is the new category vs techdebt's stock five — techdebt encodes process via hooks (PreToolUse/PostToolUse intercept tool calls). BetterAI surfaces process explicitly as retrievable rules because the agent (Claude Code, Aether, claude.ai) should see them in its context window, not have them silently enforced by a hook the agent can't introspect.

### 1.2 Frontmatter Schema (LOCKED for v1)

```yaml
---
id: <kebab-case-globally-unique>           # REQUIRED
title: <one-line title, <80 chars>         # REQUIRED
category: STANDARDS | PROCESS | PATTERNS | ARCHITECTURE | DOCUMENTATION   # REQUIRED
domain: <free-form, see v4 domain list>    # REQUIRED
applies_when:                              # OPTIONAL but recommended
  paths: ["**/*.ts", "**/*.py"]            # glob array (matched against project-relative path)
  symbols: ["fastify.post", "rescue"]      # symbol array (string-match for grep)
  intents: ["plan", "implement", "edit",   # agent-intent keywords (string-match against intent field)
            "refactor", "review", "fix"]
severity: low | medium | high              # REQUIRED (drives ranking)
check: ~                                   # OPTIONAL; kind = regex | ast-grep in v1 (no shell/ts-module per v4 revision)
fix_template: ~                            # OPTIONAL prose snippet
created: 2026-06-09                        # REQUIRED (ISO date)
last_fired: ~                              # MAINTAINED by server (null at creation)
fire_count: 0                              # MAINTAINED by server
source: ~                                  # OPTIONAL — where this rule came from (e.g., "RULES.md-original-rule-7")
related: []                                # OPTIONAL — list of related rule ids
---
```

**Body structure (LOCKED):**

```markdown
# {title}

## What this rule says
{1-2 sentence statement of the rule, agent-friendly}

## Why it matters
{Reason — what breaks if the rule isn't followed. 2-4 sentences.}

## When this applies
{Plain-prose mirror of applies_when frontmatter. Helps the agent reason about applicability.}

## What good looks like
{Concrete positive example. Code block if helpful.}

## Anti-patterns
{Specific failure modes — what the AI tends to do wrong.}
{Code block of the wrong way (annotated).}
{Code block of the right way (annotated).}

## Examples
{Optional — multiple worked examples. Each one labeled.}
```

This shape mirrors techdebt's `templates/PATTERNS.md` and `templates/STANDARDS.md` conventions (which have similar sections — Anti-patterns, When this applies, What good looks like).

### 1.4 Cross-Category Rules

Some rules belong to two categories (e.g., "Ask about requirements" is both STANDARDS — a constraint — and PROCESS — a workflow). Two policies considered:

- **Symlink:** `STANDARDS/maintainability/ask-about-requirements.md` is the canonical file; `PROCESS/methodology/ask-about-requirements.md` is a symlink. Directory listing shows both; embedding indexes both; retrieval merges. **Pros:** one source of truth; **Cons:** Docker bind-mount + symlinks on macOS/Windows historically flaky.
- **Single home + `related:` field:** rule lives in its primary category; other categories link via `related: [rule-id]`. **Pros:** no FS-specific behavior; **Cons:** category browsing misses cross-listed rules unless the UI traverses `related`.

**AUTO-DECIDE (P5 explicit > P3 pragmatic):** single home + `related:` field. Symlinks introduce a portability footgun that's outside the v1 sandbox.
