## What this rule says

Every configuration key is required and read in exactly one place (the
settings module). There are no in-code defaults: no `os.environ.get(key,
fallback)`, no `process.env.X ?? "8080"`, no "sensible default" constants
that silently apply when a key is missing. Boot validates all required keys
up front and crashes with a single error listing the exact missing key names.
The generated `.env` is fully explicit - every key present with a real value;
`.env.example` carries placeholders only.

This deliberately supersedes the softer stance of
`config-from-env-not-hardcoded` (which blessed env-var reads with defaults):
env-sourcing is still required, but the defaults are now banned too.

## Why it matters

A default is a config value that nobody chose. When `PORT ?? "8080"` kicks in
because the key was renamed, misspelled, or dropped from the deployment, the
system runs - on the wrong port, against the wrong database, with the wrong
model - and the failure surfaces far away from its cause. Explicit required
keys convert every one of those silent drifts into a boot-time crash with the
missing key named, which is a 30-second fix. Defaults also scatter effective
configuration across the codebase: to know what the system actually runs
with, you must read every call site instead of one `.env` file.

## When this applies

- Any read of environment variables or config files.
- Any settings/config module, boot sequence, or installer that writes `.env`.
- Any new key added to the configuration surface.
- Does NOT apply to test fixtures constructing a Settings object directly
  with literal values - tests choose their values explicitly by definition.

## What good looks like

One settings module, required keys, boot crash that lists everything missing
at once (not one key per crash):

```python
REQUIRED_KEYS = ("BETTERAI_PORT", "REDIS_URL", "POSTGRES_DSN", "EMBEDDING_DIM")

@classmethod
def from_env(cls, env: Mapping[str, str]) -> "Settings":
    missing = [key for key in REQUIRED_KEYS if key not in env]
    if missing:
        raise Errors.missing_config(f"missing required keys: {', '.join(missing)}")
    return cls(port=int(env["BETTERAI_PORT"]), redis_url=env["REDIS_URL"], ...)
```

## Anti-patterns

Wrong - scattered reads with silent defaults:

```python
port = int(os.environ.get("BETTERAI_PORT", "7777"))   # nobody chose 7777
model = os.environ.get("EMBED_MODEL") or "text-embedding-3-small"
```

Fixed: both keys join `REQUIRED_KEYS`, the values move to the explicit
`.env`, and the reads become `settings.port` / `settings.embedding_model`.

Wrong - crash-per-key boot, which turns a 10-key misconfiguration into ten
deploy cycles:

```python
if "REDIS_URL" not in env:
    raise Errors.missing_config("missing REDIS_URL")  # finds ONE key per boot
```

Fixed: collect all missing keys first, raise once with the full list.

## Examples

Installer side of the contract - the generated `.env` is complete, and the
example file never carries runnable values:

```bash
# .env (generated by the installer; every key explicit, chosen values)
BETTERAI_PORT=7777
REDIS_URL=redis://127.0.0.1:6379/0

# .env.example (checked in; placeholders only, never real values)
BETTERAI_PORT=<port>
REDIS_URL=<redis-url>
```

If you find yourself wanting a default "so local dev is easier", put the
value in the generated `.env` instead - same convenience, but the choice is
written down where operators look.
