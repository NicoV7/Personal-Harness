Stage 1

HTTP + bearer MCP transport

The front door. BetterAI speaks the Model Context Protocol over a streamable HTTP transport rather than stdio, so any networked agent can reach it. Every inbound call must present a bearer token; authentication runs before parsing or dispatch, so an unauthenticated request never reaches the orchestrator or touches the corpus.

In raw HTTP request  ·  Out validated retrieve_context call

Stage 2

Connection limiter

A semaphore over in-flight retrievals. It caps concurrency so a sudden burst of agents queues or backs off cleanly rather than oversubscribing CPU and blowing latency for everyone. The cap is the difference between a graceful slowdown and a thundering-herd collapse.

In authenticated call  ·  Out admitted call (or back-off)

Stage 3

LRU cache

Before any scoring runs, the orchestrator checks an LRU cache keyed on a hash of the task text plus the detected repo scope. Identical questions from the same scope are common in an agent fleet, so a hit short-circuits the whole pipeline and returns ranked context immediately. Entries carry a TTL so corpus edits propagate without a restart.

In task + scope key  ·  Out cached context (hit) or miss

Stage 4 · core

Retrieval orchestrator

The heart of BetterAI. On a cache miss it runs two retrievers in parallel: a grep pass for exact lexical matches (rule IDs, API names, file globs) and a MiniLM embedding pass for semantic similarity to the task. The two score sets are merged with a weighted combination, deduplicated, and ranked; the orchestrator returns the top-k documents. Hybrid scoring is deliberate — grep alone misses paraphrases, embeddings alone miss precise identifiers. 

lex   = grep(task, corpus)            // exact hits
sem   = embed(task) · embed(corpus)  // MiniLM cosine
score = w_lex * lex + w_sem * sem    // weighted merge
return rank(score)[:k]

In task + scope  ·  Out ranked top-k rules / skills / memories

Stage 5

Repo-scope detection

Inspects the calling repository — its languages, manifests, and layout — to infer a domain scope (frontend, backend, infra, and so on). The scope narrows the candidate set so a CSS task never gets database migration rules, and it becomes part of the cache key. Cheap, deterministic, and high-leverage for precision.

In repo signals  ·  Out scope label feeding the scorer + cache

Stage 6

Markdown corpus reader

The corpus is plain markdown — one file per rule, skill, or memory, with a small typed header. The reader parses those files into scorable documents (id, category, severity, body) that both retrievers consume. Authoring a new rule is just dropping in a markdown file; no schema migration, no rebuild.

In markdown files  ·  Out parsed rule / skill / memory documents

Stage 7

JSONL audit log

Every retrieval appends one JSON line: timestamp, query, detected scope, matched IDs, and scores. Because it’s append-only JSONL it’s trivially grep-able and replayable — you can prove what an agent was told and why, and reconstruct any decision after the fact. It’s also how behavioral rules (like “verify uncertain facts before deciding”) become observable. 

{"ts":"2026-06-10T12:01:44Z","scope":"frontend",
 "query":"portfolio page accessibility",
 "matched":["alt-text-required","no-inline-styles",
            "responsive-viewport-meta"],"k":4}

In retrieval result  ·  Out one appended JSONL record

Out of band

CLI

A thin command-line front end over the same orchestrator. It lets an engineer run a retrieval locally — to debug scoring, preview what an agent would get, or author and test corpus rules — without standing up an MCP client. Same code path as the server, so what you see in the terminal is what the fleet sees on the wire. 

$ betterai retrieve "build a portfolio page" --repo ./site
→ matched: alt-text-required, no-inline-styles,
           responsive-viewport-meta  (scope: frontend)
