Reference

Components

Each part of BetterAI, what it does, and how it connects to the rest of the system.

Edge & safety

HTTP + Bearer MCP transport

The transport speaks the Model Context Protocol over HTTP. Every request must carry a valid bearer token; unauthenticated calls are rejected before they reach any retrieval logic. It exposes one tool to clients — retrieve_context — and is the only network surface the server presents.

  • Inbound from Coding agents and the CLI
  • Hands off to Connection limiter, then the orchestrator
  • Auth Bearer token on every request
Edge & safety

Connection limiter

A concurrency gate that caps how many retrievals run at once. Because the embedding model is the most expensive resource in the system, the limiter sheds or queues excess load so latency stays predictable under bursty agent traffic rather than collapsing.

  • Protects The MiniLM embedding step
  • Behavior Bounds in-flight requests; rejects/queues overflow
Core retrieval

Retrieval orchestrator

The brain of the server and the implementation behind retrieve_context. For each query it: checks the LRU cache, applies repo-scope detection, invokes the hybrid scorer over the corpus, assembles the ranked result, and writes an audit record. It is the only component that talks to all the others.

  • Reads LRU cache, repo scope, scored corpus documents
  • Writes JSONL audit log, LRU cache (on miss)
  • Returns Ranked rules / skills / memories to the transport
Core retrieval

Hybrid scorer (grep + MiniLM)

Combines two complementary signals. A grep pass finds exact and substring lexical matches — high precision, great for identifiers and verbatim phrases. A MiniLM embedding pass finds semantic neighbors — high recall, great for paraphrases. The two scores are normalized and merged into a single ranking, with weighting tuned so short, specific queries lean lexical and broad queries lean semantic.

  • Inputs Query text + candidate documents from the corpus
  • Signals Lexical (grep) + semantic (MiniLM embeddings)
  • Output A single normalized, ranked list
Storage & I/O

Markdown corpus reader

Loads the knowledge base from markdown files on disk and turns it into a set of queryable documents. The corpus is organized into three kinds of entries — rules (how to behave), skills (how to do things), and memories (what happened before). The reader handles parsing, front-matter, and presenting clean text to the scorer.

  • Source Markdown files (rules, skills, memories)
  • Consumed by The hybrid scorer
Storage & I/O

LRU cache

A least-recently-used cache keyed by the normalized query (and scope). On a hit, the orchestrator returns the prior result without re-running grep or the embedding model — the cheapest possible path. On a miss, the freshly scored result is written back and the oldest entry is evicted.

  • Key Normalized query + repo scope
  • Eviction Least-recently-used
Storage & I/O

JSONL audit log

An append-only log with one JSON object per line per request. Each record captures the query, the documents that matched, their scores, whether the cache was hit, and the latency. This makes retrievals fully replayable — you can answer "exactly what context did the agent receive?" long after the fact.

  • Format JSON Lines, append-only
  • Captures Query, matches, scores, cache hit/miss, latency
Edge & safety

Repo-scope detection

Determines which repository the request is associated with and constrains retrieval to rules and memories relevant to that repo. This keeps an agent working in one project from being handed context that only applies to another.

  • Effect Filters corpus candidates by repository
  • Used by The orchestrator before scoring
Client / entrypoint

CLI

A command-line entrypoint for running the server in Docker and for querying the corpus directly without going through an agent. Handy for debugging the scorer, seeding the cache, or inspecting what a given query would return.

  • Uses Same orchestrator path as the MCP transport
  • For Local runs, debugging, manual queries
